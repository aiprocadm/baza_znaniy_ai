# Keyless Local Stack as Default — Design

**Date:** 2026-06-08
**Status:** Design approved (brainstorming). Awaiting spec review → implementation plan.
**Scope owner:** solo side-project (10-20 h/week)
**Parent strategy:** subordinate to [2026-05-22-project-vision-design.md](2026-05-22-project-vision-design.md)

> Цель документа — спроектировать «локальный keyless-стек как дефолт» для self-hosted
> установки (бизнес-tier **B**), архитектурно так, чтобы тот же инференс-движок
> переиспользовался в будущем облачном tier **C** («зарегистрировался и попробуй»),
> **не строя C в рамках этой работы** (YAGNI).

---

## 1. Problem and motivation

Сейчас свежая self-hosted установка **не работает из коробки без ручной настройки**.
Без ключа внешнего провайдера и без явного `KB_LLM_PROVIDER=gguf` выбор LLM проваливается
в `None` → extractive-заглушку, а эмбеддер по умолчанию — `hashing` (семантический мусор).
Итог: клиент упирается в «скачай Ollama / достань API-ключ» — это и есть стоимость
онбординга, которая ломает продажу self-hosted продукта.

Боль, сформулированная заказчиком: *«хочу, чтобы клиенту не доставляла сложностей работа
с проектом — поставил и пользуешься, без дополнительных скачиваний нейронок и API»*.

**Бизнес-рамка (зафиксирована в брейнсторме):**
- **Сейчас** строим B (self-hosted без трения).
- **В будущем** добавляем C (облачный two-tier поверх) — отдельный крупный этап, не в этом scope.
- Кодовая база уже имеет две дорожки под два tier'а: `/api/kb/*` (MVP → self-hosted продукт B)
  и `/api/v1/*` (multi-tenant → облачный tier C). Сливать их **нельзя** (см. `docs/architecture.md`);
  они и есть естественное разделение двух tier'ов.

## 2. Goal / non-goals

**Goal:** `docker compose up` на свежей установке даёт работающий RAG (поиск + ответы
с цитатами) **без единого ключа и без ручной возни**; внешние API, GPU и облако — опциональные
апгрейды поверх того же кода.

**Non-goals (этот scope их НЕ включает):**
- Облачный multi-tenant tier C, биллинг, регистрация аккаунтов, изоляция тенантов.
- Обучение собственной foundation-модели с нуля (рассмотрено и отклонено: месяцы и
  сотни тысяч $; нужное «дообучение под клиента» уже покрывается LoRA-пайплайном).
- Переписывание векторного слоя, reranker'а, Docling-парсинга, UI или роутов.
- Расширение списка внешних LLM-провайдеров.

## 3. Architecture — «works by default, upgrade is optional»

Локальный стек становится **основой**, а не fallback последней надежды.

**Новая цепочка выбора LLM** (`select_provider`):
```
KB_LLM_PROVIDER задан явно?            → используем его            (как сейчас)
  ↓ нет
есть ключ внешнего провайдера?         → используем внешний        (как сейчас)
  ↓ нет
локальная GGUF-модель есть на диске?   → ИСПОЛЬЗУЕМ ЕЁ             ← НОВОЕ keyless-звено
  ↓ нет
extractive-заглушка                    → реально последний резерв  (как сейчас)
```

**Эмбеддер** — дефолт меняется `hashing` → bundled **multilingual-e5-small**
(~120 МБ, 384 dim, асимметричные префиксы `query:`/`passage:`). Hashing остаётся
только для unit-тестов и как явный фоллбэк.

**Два слоя конфигурации:**
- **Дефолт (zero-config, под compliance-moat):** GGUF (Qwen2.5-3B Q4) + e5-small **зашиты
  в образ** → работает офлайн / в air-gapped сети без интернета.
- **Опциональные профили (compose overrides, только env, ноль кода):**
  - `api` — внешний ключ, без локальной модели (лёгкий образ).
  - `gpu` — 7B-модель, CUDA-слои.
  - `airgap` — алиас дефолта (явное имя для CISO-документации).

**Граница:** вся логика — в `select_provider` + новом резолвере эмбеддера. Роуты
(`kb_mvp.py`) и UI **не меняются** — они дёргают абстракцию `generate()`/`embed()`.

**Forward-compat к tier C:** тот же `select_provider`/`GgufEvalProvider`/`e5` работает,
если запустить его централизованно на сервере; разница только в multi-tenant обёртке,
которую уже даёт `/api/v1/*`. Поток данных B = поток данных C.

## 4. Components (responsibility · files · deps)

| # | Компонент | Файл | Меняем код? |
|---|-----------|------|-------------|
| 1 | Выбор LLM: GGUF как keyless-звено авто-цепочки | `app/services/kb_llm.py` (`select_provider`, ~L504) | да (рантайм) |
| 2 | Выбор эмбеддера: e5-small по умолчанию | `app/services/kb_embeddings.py` | да (рантайм) |
| 3 | Провижининг модели + манифест | `scripts/download_model.py`, `models/model_manifest.json`, `Dockerfile` (build-arg `BUNDLE_MODEL`) | конфиг/скрипты |
| 4 | Профили запуска | `compose.yml` + `compose.api.yml` / `compose.gpu.yml` | конфиг |
| 5 | Детект железа / авто-деградация (совет, не решение) | новый `app/services/hardware_probe.py` | да (изолированный) |
| 6 | Preflight-лог активного режима при старте | lifespan / `scripts/dev_server_mvp.py` | мелкий |

Из шести — четыре конфиг/скрипты/доки, два трогают рантайм. Это «дотягивание»,
а не «строительство»; маленький diff в горячем пути → низкий риск регрессий.

**Детали:**
- (1) `_build_gguf_provider` (kb_llm.py ~L477) уже корректен (проверяет файл, логирует
  подсказку, знает дефолтный путь `./models/qwen2.5-3b-instruct-q4_k_m.gguf`) — переиспользуем
  как есть, просто включаем в авто-цепочку перед возвратом `None`.
- (2) ST-эмбеддер уже протестирован (`tests/test_kb_embeddings_st.py`) — довести до прод-класса
  рядом с `HashingEmbedder`/`OpenAICompatibleEmbedder`; **обязательно** использовать
  `app/retriever/e5.py:e5_prefix` для асимметрии.
- (5) `hardware_probe` намеренно отделён от `select_provider`: детект железа — «совет»,
  выбор — «решение». Разделение нужно для детерминированного тестирования.

## 5. Data flow

**Старт (lifespan, один раз):**
```
hardware_probe (RAM/cores/CUDA) → лог-совет (не блокирует)
select_embedder() → e5-small если веса есть, иначе hashing
select_provider() → внешний ключ → иначе GGUF если файл есть → иначе extractive
preflight-лог: "LLM=gguf(qwen2.5-3b) · Embedder=e5-small · Mode=bundled/airgap"
кэш выбранных в app.state
```

**Ingest:** `файл → Docling → чанки → embedder.embed(passage) → вектор`.
Единственное изменение — реальный e5 вместо hashing.

**Запрос:** `query → embedder.embed(query) → top-k поиск → BGE reranker → контекст →
provider.generate() → ответ+цитаты → SSE`. `provider` = GGUF локально или внешний API;
интерфейс идентичен → роуты/UI не замечают разницы.

Reranker (BGE) остаётся в потоке и частично компенсирует слабость мелкой 3B-модели:
на CPU-сетапе хороший контекст важнее размера LLM.

## 6. Error handling, degradation, migration

Сквозной принцип: **fail loud, never silent.** Проект уже обжигался на тихих
деградациях (hashing-эмбеддер, grep-фоллбэк вектор-стора) — каждый деградированный
режим обязан оставлять видимый след.

1. **Миграция эмбеддера (главный риск).** Смена дефолта hashing→e5 меняет размерность.
   Записываем сигнатуру эмбеддера (`name+dim`) в метаданные индекса (SQLite). При старте
   сверяем:
   - свежий индекс → пишем сигнатуру e5, стартуем без шума;
   - индекс на hashing + активен e5 → **жёсткий стоп** с инструкцией
     `kb-cli reindex --embedder e5-small` (или явный возврат `KB_EMBEDDINGS_BACKEND=hash`).
     **Не авто-reindex** — он мог бы внезапно сжечь часы на большом корпусе.
2. **Мало RAM (профиль «железо разное»).** `hardware_probe` сравнивает потребность
   (~3-4 ГБ) с доступной RAM → видимый WARNING + конкретный совет (профиль `api`),
   но запуск продолжается. Никаких тихих OOM.
3. **GGUF отсутствует/битый.** Наличие проверяется уже сейчас; добавляем быструю
   SHA256-сверку с манифестом (битая закачка → понятная ошибка вместо креша llama.cpp).
   Нет файла и нет ключей → extractive + явный лог «деградированный режим, потому что …».
4. **Лёгкий образ (`api`) без ключа.** Ошибка конфигурации **при старте** с указанием
   переменной, а не рантайм-сюрприз на первом запросе.

## 7. Testing (TDD, со стабами)

1. **`select_provider` (unit, без сети):** нет ключей+файл есть → `gguf`; нет ключей+нет файла
   → `None`; есть ключ+есть файл → внешний (приоритет сохранён); `KB_LLM_PROVIDER=gguf` → gguf
   (регресс). GGUF мокается через `inner=`.
2. **Эмбеддер (unit):** backend не задан + веса есть → e5; нет весов → hashing; префиксы
   `query:`/`passage:` применяются корректно.
3. **Guard сигнатуры (unit, ключевой по риску):** свежий → пишет+стартует; рассогласование
   → жёсткая ошибка с инструкцией; совпадение → молча стартует.
4. **`hardware_probe` (unit):** RAM хватает → без warning; мало → warning, но `select_provider`
   не падает. RAM/CUDA мокаются.
5. **Профили (smoke):** расширить `tests/test_env_example.py` — каждый профиль даёт валидную
   непротиворечивую конфигурацию.
6. **Стабы:** все новые тесты поверх `tests/stubs/` (`llama_cpp`, `sentence_transformers`);
   реальная загрузка модели → `@pytest.mark.integration`, не `skip`.

**Гло­бальное состояние:** выбор кэшируется в `app.state` → фикстуры **обязаны** сбрасывать
кэш между тестами (иначе «выбран gguf» протечёт в тест «выбран внешний» — известная ловушка
полного прогона vs одиночного).

## 8. Risks

- **Размер образа при `BUNDLE_MODEL=true`:** дефолтный образ становится ~3-5 ГБ. Принято
  осознанно ради air-gap/compliance; профиль `api` даёт лёгкую альтернативу.
- **Качество 3B по-русски:** приемлемое, не топовое. Митигируется reranker'ом и опцией
  апгрейда на 7B/внешний API.
- **Лицензии моделей:** зафиксировать `license` в манифесте для Qwen2.5-3B и e5-small
  (нужно для коммерческой поставки клиенту).

## 9. Open questions

Нет на момент финального дизайна — все развилки (hybrid-провижининг, жёсткий стоп при
рассогласовании эмбеддера, hardware_probe как отдельный модуль) зафиксированы в брейнсторме.
