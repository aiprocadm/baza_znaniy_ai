# Own Reranker via Distillation (`kbai-reranker-ru`) — Design

**Date:** 2026-06-10
**Status:** Design approved (brainstorming). Awaiting spec review → implementation plan.
**Scope owner:** solo side-project (10-20 h/week)
**Parent strategy:** subordinate to [2026-05-22-project-vision-design.md](2026-05-22-project-vision-design.md); builds on [2026-06-08-keyless-local-stack-default-design.md](2026-06-08-keyless-local-stack-default-design.md) (tier B) and [2026-06-06-hybrid-eval-corpus-design.md](2026-06-06-hybrid-eval-corpus-design.md) (данные/гейт).

> Цель документа — спроектировать **собственную маленькую модель** проекта:
> дистиллированный русскоязычный cross-encoder реранкер (~29M параметров),
> который вставляется в уже существующий реранк-слот и делает реранкинг
> доступным на CPU в tier B. Это ответ на стратегический запрос «своя
> нейросеть/модель» в реалистичном для соло-проекта объёме: LLM-с-нуля
> отклонён ранее (см. memory/решение 2026-06-08), маленькая
> специализированная сеть — выбранный путь.

---

## 1. Problem and motivation

1. **Реранк-слот в коде есть, но мёртв для tier B.** `app/retriever/rerank.py` +
   `app/services/kb_rerank.py` реализуют cross-encoder реранкинг (env-контракты
   `RERANK_*` для `/api/v1/*` и `KB_RERANK_*` для `/api/kb/*`), но дефолтная
   модель `BAAI/bge-reranker-v2-m3` — ~568M параметров. На CPU клиента
   (tier B, без GPU) она не укладывается в бюджет латентности, поэтому
   реранкинг по умолчанию выключен и фактически недоступен в keyless-стеке.
2. **Стратегический запрос — «своя модель».** Заказчику нужен актив
   «у нас собственная нейросеть» (продукт/позиционирование). Претрейн LLM
   с нуля отклонён (месяцы, $100k+); дистиллированная маленькая модель
   даёт «свои веса» за дни работы и при этом закрывает реальную продуктовую
   дыру из п.1, а не существует ради галочки.
3. **Данные для обучения уже есть в репо.** Публичный eval-корпус
   (`data/eval/corpus_public/`, 9 RU-документов), замороженные bge-m3
   векторы (`frozen_bge-m3.npz`) для майнинга hard negatives, конвейер
   синтетических QA (W1, `app/services/synthetic_qa.py`) и eval-гейт
   (`data/eval/golden_public.jsonl` + `scripts/eval_rag.py`) как судья.

## 2. Goal / non-goals

**Goal:** обучить дистилляцией собственный cross-encoder реранкер
(~29M параметров, рабочее имя `kbai-reranker-ru`), который:

- на `golden_public` даёт качество поиска **строго лучше**, чем пайплайн без
  реранкера (метрики существующего eval-харнесса) — это релизный гейт;
- переранжирует 20 кандидатов на CPU за ~100–200 мс (иначе бессмыслен в tier B);
- подключается в существующий слот через `KB_RERANK_MODEL`/`RERANK_MODEL`
  **без нового продакшн-кода** (допустима точечная правка для загрузки весов
  из локальной папки, если HF-путь не сработает из коробки);
- воспроизводим: скрипты генерации данных и обучения коммитятся в репо.

**Non-goals (этот scope их НЕ включает):**

- Свой эмбеддер (замена e5-small/bge-m3) — отдельный будущий спек;
  пайплайн дистилляции проектируем переиспользуемым, но эмбеддер не делаем.
- Пер-клиентский тюнинг реранкера на корпусе клиента (будущий moat рядом с LoRA).
- Публикация весов на Hugging Face — **решение: пока приватно**, веса
  распространяются только в бандле продукта (как `BUNDLE_MODEL` сейчас).
- GPU-инфраструктура и внешние тренинг-сервисы — **решение: обучение локально
  на CPU** (совместимо с принципом keyless/локально).
- Включение реранкера по умолчанию в tier B — отдельный шаг после
  прохождения гейта, не часть этого спека (но спек обязан его не блокировать).

## 3. Архитектура решения

### 3.1 Модель

- **База ученика:** `cointegrated/rubert-tiny2` (~29M параметров, RU-центричная,
  CPU-дружелюбная). Поверх — стандартная голова cross-encoder
  (sequence classification, 1 логит = скор релевантности пары).
- **Учитель:** `BAAI/bge-reranker-v2-m3` — уже дефолт реранк-слота
  (`DEFAULT_MODEL_NAME` в `app/services/kb_rerank.py`), т.е. сравнение
  ученик-vs-учитель честно меряет «что мы теряем относительно тяжёлого пути».

### 3.2 Данные (дистилляция, без ручной разметки)

Конвейер (новый скрипт в `scripts/`, по образцу существующих eval-скриптов):

1. **Запросы:** существующий synthetic-QA генератор (W1) по документам
   `data/eval/corpus_public/`. Запросы из `golden_public.jsonl` в обучение
   **не попадают** — golden только для оценки (см. п.3.4 про анти-утечку).
2. **Кандидаты:** для каждого запроса — топ-N соседей по замороженным
   bge-m3 векторам (`frozen_bge-m3.npz`); нерелевантные соседи = hard negatives.
3. **Метки:** учитель скорит каждую пару (запрос, фрагмент); ученик учится
   воспроизводить скоры (margin-MSE — стандартный рецепт дистилляции реранкеров).
4. Объём цели: ~50–100k пар. Датасет — generated-артефакт в `var/`
   (не коммитится; коммитятся скрипты + конфиг генерации).

### 3.3 Обучение

- `sentence-transformers` CrossEncoder-тренинг, локально на CPU
  (стек py 3.13: torch CPU + sentence-transformers уже установлены).
  Оценка: часы на 29M-модели при 50–100k пар.
- Артефакт: HF-формат каталог весов в `var/models/kbai-reranker-ru/`
  (вне git), с manifest/sig по образцу остальных generated-артефактов.

### 3.4 Оценка и анти-утечка

- **Трёхстороннее сравнение** на `golden_public.jsonl` через существующий
  eval-харнесс: (a) без реранкера, (b) учитель bge-reranker-v2-m3,
  (c) ученик `kbai-reranker-ru`. Гейт CI не меняется — он и есть судья.
- **Анти-утечка:** запросы из `golden_public.jsonl` (и приватного golden)
  исключаются из обучающего датасета — фильтр по точному совпадению и по
  doc+chunk парам golden. Обучаемся на synthetic-QA запросах, оцениваемся
  на golden. Корпус документов общий — это допустимо (реранкер учится
  на домене, golden меряет обобщение на новые запросы).

### 3.5 Интеграция

- Подключение через существующие env: `KB_RERANK_MODEL=<локальный путь>`
  (MVP) / `RERANK_MODEL` (v1). `CrossEncoder(model_name)` в
  sentence-transformers принимает локальный путь — ожидаем ноль правок;
  если выяснится иное, допускается минимальная правка резолва пути.
- Дистрибуция: приватно, в бандле продукта тем же механизмом, что
  `BUNDLE_MODEL` (см. keyless-stack спек), отдельным шагом после гейта.

## 4. Критерии успеха (must-pass)

1. На `golden_public` метрики поиска с учеником **> без реранкера**:
   минимум +2 п.п. `hit@5` или +0.02 `mrr@5` (метрики `app/eval/metrics.py`);
   точные пороги план фиксирует после замера базлайна, но «не хуже ни по
   одной из hit@5/recall@5/mrr@5» — обязательное условие.
2. Латентность: реранк 20 кандидатов на CPU этой машины ≤ ~200 мс (замер
   в составе eval-скрипта, не микробенчмарк на глаз).
3. Воспроизводимость: `scripts/` позволяют повторить генерацию данных и
   обучение с фиксированным seed; конфиг и версии зафиксированы.
4. Стретч (не гейт): ученик достигает ≥90% качества учителя на golden_public.

## 5. Риски и митигации

| Риск | Митигация |
|---|---|
| Ученик 29M не дотягивает до учителя 568M | Гейт — «лучше, чем без реранкера», не «как учитель»; стретч-метрика отдельно. Если не лучше — модель не релизится, пайплайн остаётся активом для итераций. |
| Шумные synthetic-QA запросы | Метки ставит учитель (не генератор) — дистилляция сглаживает шум; ручная проверка случайной выборки пар глазами перед обучением. |
| CPU-обучение окажется слишком долгим | 29M + 50–100k пар — часы; если упрёмся — режем датасет/эпохи, фиксируем в плане dead-man критерий. Длинные запуски — detached (см. memory: detached-long-runs). |
| Утечка golden в трейн | Явный фильтр анти-утечки (п.3.4) + проверка пересечения в самом скрипте генерации (assert, не договорённость). |
| Корпус из 9 документов мал для обобщения | Для v1 цель — домен RU-документов проекта, не универсальная модель; расширение корпуса — отдельная итерация. Честно фиксируем ограничение в model card. |

## 6. Decisions log (из брейнсторма 2026-06-10)

- «Своя модель» = **маленькая специализированная сеть с нуля/дистилляцией**,
  не LLM (LLM-с-нуля отклонён 2026-06-08), не учебный проект.
- Роль модели: **реранкер** (вариант A) — выбран против эмбеддера (B)
  и классификатора запросов (C); B остаётся кандидатом на переиспользование
  пайплайна дистилляции.
- Обучение: **локально на CPU**, без Colab/Kaggle.
- Дистрибуция весов: **пока приватно** (бандл продукта), публикация на HF —
  отложенное решение.
